description: "REST API reference for the organizational hierarchy: manage clients, plants, sectors, and locations that structure dispenser deployments."
---

HDB Service uses a four-level organizational hierarchy: **Client → Plant → Sector → Location**. A client represents a business account. Each client has one or more plants (physical sites). Plants can be subdivided into sectors (logical zones), and each sector contains locations — the precise spots where water dispensers are installed. Locations carry a globally unique, user-defined ID that is printed on QR codes attached to each dispenser mounting point.

All endpoints require an authenticated session. Access is automatically scoped by the RBAC system: `CLIENT_RESPONSIBLE` and `CLIENT_REQUESTER` roles can only read data belonging to their own client or assigned plants, and cannot create or modify records.

## GET /api/clients

Returns all active clients visible to the authenticated user. `CLIENT_RESPONSIBLE` and `CLIENT_REQUESTER` users receive only their own client record. Each client object includes a `_count` summary with the number of associated plants and users.

**Required permission:** `clients:read`

<ResponseField name="id" type="string">Client ID.</ResponseField>
<ResponseField name="nombre" type="string">Client display name.</ResponseField>
<ResponseField name="email" type="string">Contact email (nullable).</ResponseField>
<ResponseField name="telefono" type="string">Contact phone number (nullable).</ResponseField>
<ResponseField name="direccion" type="string">Physical address (nullable).</ResponseField>
<ResponseField name="active" type="boolean">Whether the client is active.</ResponseField>
<ResponseField name="createdAt" type="string">ISO 8601 creation timestamp.</ResponseField>
<ResponseField name="_count" type="object">
  <Expandable title="_count fields">
    <ResponseField name="plants" type="number">Number of plants belonging to this client.</ResponseField>
    <ResponseField name="users" type="number">Number of users associated with this client.</ResponseField>
  </Expandable>
</ResponseField>

```bash
curl -X GET "https://your-domain.com/api/clients" \
  -H "Cookie: sb-access-token=<token>"
```

**Error responses:**

| Status | Description |
|--------|-------------|
| `401` | Not authenticated. |
| `403` | Insufficient permission (`clients:read` required). |
| `500` | Internal server error. |

## POST /api/clients

Creates a new client. The `nombre` field is required; all other fields are optional. Creation is idempotent via the `Idempotency-Key` header — sending the same key twice returns the first response without creating a duplicate. The action is written to the audit log.

**Required permission:** `clients:write`

<ParamField body="nombre" type="string" required>
  Client display name. Cannot be blank.
</ParamField>

<ParamField body="email" type="string">
  Contact email address.
</ParamField>

<ParamField body="telefono" type="string">
  Contact phone number.
</ParamField>

<ParamField body="direccion" type="string">
  Physical address.
</ParamField>

<ResponseField name="id" type="string">Newly created client ID.</ResponseField>
<ResponseField name="nombre" type="string">Client display name.</ResponseField>
<ResponseField name="email" type="string">Contact email (nullable).</ResponseField>
<ResponseField name="telefono" type="string">Contact phone (nullable).</ResponseField>
<ResponseField name="direccion" type="string">Physical address (nullable).</ResponseField>
<ResponseField name="active" type="boolean">Always `true` on creation.</ResponseField>
<ResponseField name="createdAt" type="string">ISO 8601 creation timestamp.</ResponseField>

```bash
curl -X POST "https://your-domain.com/api/clients" \
  -H "Content-Type: application/json" \
  -H "Cookie: sb-access-token=<token>" \
  -d '{
    "nombre": "Acme Corporation",
    "email": "ops@acme.com",
    "telefono": "+54 11 4000-0000",
    "direccion": "Av. Corrientes 1234, Buenos Aires"
  }'
```

**Error responses:**

| Status | Description |
|--------|-------------|
| `400` | `nombre` is missing or blank. |
| `401` | Not authenticated. |
| `403` | Insufficient permission (`clients:write` required). |
| `500` | Internal server error. |

## POST /api/locations

Creates a new location. The `id` field is the globally unique location code printed on QR codes — you may supply your own (e.g., `"A-102"`) or omit it to have the server auto-generate a sequential ID in the format `LOC-001`, `LOC-002`, etc. The `plantId` and `nombre` are always required. Creation is idempotent via `Idempotency-Key`. The action is recorded in the audit log.

**Required permission:** `locations:write`

<ParamField body="id" type="string">
  User-defined globally unique location ID (e.g., `"SECTOR-A-101"`). If omitted or blank, the server auto-generates a sequential `LOC-NNN` ID.
</ParamField>

<ParamField body="plantId" type="string" required>
  ID of the parent plant.
</ParamField>

<ParamField body="nombre" type="string" required>
  Location display name. Cannot be blank.
</ParamField>

<ParamField body="sectorId" type="string">
  ID of the sector this location belongs to.
</ParamField>

<ParamField body="piso" type="string">
  Floor identifier (e.g., `"2"`, `"PB"`, `"Mezzanine"`).
</ParamField>

<ParamField body="area" type="string">
  Area within the sector (e.g., `"Sala de reuniones"`).
</ParamField>

<ParamField body="descripcion" type="string">
  Free-text description for technicians.
</ParamField>

<ResponseField name="id" type="string">Location ID (user-supplied or auto-generated).</ResponseField>
<ResponseField name="plantId" type="string">Parent plant ID.</ResponseField>
<ResponseField name="sectorId" type="string">Sector ID (nullable).</ResponseField>
<ResponseField name="nombre" type="string">Location display name.</ResponseField>
<ResponseField name="piso" type="string">Floor (nullable).</ResponseField>
<ResponseField name="area" type="string">Area (nullable).</ResponseField>
<ResponseField name="descripcion" type="string">Description (nullable).</ResponseField>
<ResponseField name="active" type="boolean">Always `true` on creation.</ResponseField>
<ResponseField name="createdAt" type="string">ISO 8601 creation timestamp.</ResponseField>

```bash
curl -X POST "https://your-domain.com/api/locations" \
  -H "Content-Type: application/json" \
  -H "Cookie: sb-access-token=<token>" \
  -d '{
    "plantId": "plant_xyz",
    "sectorId": "sector_admin",
    "nombre": "Recepción Principal",
    "piso": "PB",
    "area": "Hall de Entrada"
  }'
```

<Warning>
  Location IDs are **globally unique** across all plants and clients. If you supply a custom `id` that already exists anywhere in the system, the request returns `409 Conflict`.
</Warning>

**Error responses:**

| Status | Description |
|--------|-------------|
| `400` | `plantId` or `nombre` is missing. |
| `409` | Supplied `id` already exists in the system. |
| `401` | Not authenticated. |
| `403` | Insufficient permission (`locations:write` required). |
| `500` | Internal server error. |

## DELETE /api/sectors/[id]

Soft-deletes a sector by setting `active: false`. The sector record is retained in the database. The action is recorded in the audit log.

**Required permission:** `sectors:write`

<ResponseField name="id" type="string">Sector ID.</ResponseField>
<ResponseField name="active" type="boolean">Always `false` after a successful delete.</ResponseField>
<ResponseField name="updatedAt" type="string">ISO 8601 timestamp of deactivation.</ResponseField>

```bash
curl -X DELETE "https://your-domain.com/api/sectors/sector_admin" \
  -H "Cookie: sb-access-token=<token>"
```
